**Template 1: Episodic Summarisation Prompt (Haiku/Sonnet)**
- Purpose: compress raw episodic events into meaningful day-level summaries grouped by context
- Input structure: how raw events are formatted for the LLM (timestamp, signal type, content, importance score)
- System prompt: set the role, constraints, output format
- User prompt: the actual event data injection format
- Output schema: structured JSON with summary blocks, key events, anomaly flags

- Quality criteria: what makes a good vs bad summarisation
- Failure modes: what goes wrong and how to detect it
- Token budget: estimated input/output tokens per call

**Template 2: First-Order Pattern Extraction Prompt (Sonnet)**
- Purpose: identify observable patterns from today's summarised events + recent history
- Input structure: today's episode summary + recent episode summaries (7 days) + existing pattern library
- System prompt: define what a first-order pattern is, provide the pattern taxonomy, calibrate confidence
- Memory injection format: how existing patterns are provided for deduplication and update
- Output schema: JSON array of detected patterns with evidence chains, confidence scores, category tags
- Few-shot examples: 2-3 examples showing pattern detection from episode summaries
- Calibration instructions: "Only report patterns supported by at least N observations" — what is N per category?
- How to handle novel patterns (ones not in the taxonomy)
- Token budget

**Template 3: Second-Order Insight Synthesis Prompt (Opus)**
- Purpose: cross-reference first-order patterns to generate named insights
- Input structure: today's detected patterns + full pattern library + current semantic model + existing insight library
- System prompt: define what a second-order insight is, provide examples of the depth expected, set confidence requirements
- Memory injection format: how the executive's semantic model is presented (key facts, trait scores, historical context)
- Output schema: JSON with insight objects (title, description, evidence chain, confidence, actionability, counter-argument)
- Few-shot examples: 2-3 examples showing pattern combination → insight generation at the quality level expected
- Self-critique requirement: every insight must include the strongest argument against it
- Compounding instruction: "Build on existing insights rather than regenerating surface-level observations. Reference and deepen insights from previous reflections."
- Token budget

**Template 4: Rule Generation Prompt (Opus)**
- Purpose: convert insights into procedural rules that constrain system behaviour
- Input structure: new and updated insights + existing active rules + executive override history + rule effectiveness scores
- System prompt: define what a rule is, emphasise that rules constrain the SYSTEM not the executive, provide the rule schema, include conflict detection instructions
- Output schema: JSON with candidate rules (trigger condition, recommended action, evidence source, confidence)
- Few-shot examples: 2-3 examples showing insight → rule generation
- Negative examples: 2-3 examples of rules that should NOT be generated (rules that try to control the executive, rules without sufficient evidence, vague rules)
- Conflict detection: "Check each candidate rule against the existing rule set and flag potential conflicts"
- Token budget

**Template 5: Model Update Prompt (Sonnet)**
- Purpose: update the semantic memory store with new facts derived from tonight's reflection
- Input structure: current semantic model + tonight's insights and patterns + proposed changes
- System prompt: conservative update policy — only update facts with sufficient evidence, never overwrite high-confidence facts with low-confidence updates
- Output schema: JSON diff of semantic model (additions, updates with old/new values and confidence, no deletions without strong counter-evidence)
- Version control: output includes a model version number and change summary
- Token budget

**Template 6: Morning Intelligence Preparation Prompt (Sonnet)**
- Purpose: compile tonight's reflection outputs into a morning session brief
- Input structure: new insights, updated patterns, active rules, calendar for tomorrow, pending tasks, relationship alerts
- System prompt: prioritise ruthlessly — the executive has 90 seconds. Lead with the most important intelligence. Adapt tone based on procedural rules about this executive's preferences.
- Output schema: structured brief with priority-ordered intelligence items, each with a one-line summary and optional detail expansion
- Personalisation: how to inject the executive's communication preferences (direct vs diplomatic, data-heavy vs narrative, etc.)
- Token budget
