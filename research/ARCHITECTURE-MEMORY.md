## 1. Memory Tier Architecture

### Design Rationale

A 5-tier composite architecture — not 3-tier (episodic/semantic/procedural), which loses the intermediate abstraction levels required for compounding intelligence. The tiers form a strict hierarchy: each tier's content is synthesised from the tier below it. Intelligence compounds because higher tiers contain representations that are structurally impossible without the lower tiers having been built and validated first.

The composition draws from: Zep's bi-temporal fact model (contradiction handling), Stanford Generative Agents' importance scoring (retrieval), MAGMA's intent-aware retrieval (query routing), ICLR 2026 neuroscience precision scalar (trait stability), and SleepGate's conflict tagger (consolidation).

### Tier Definitions

| Tier | Name | Contents | Token Budget | Volume | Generated By | Embedding Model |
|------|------|----------|-------------|--------|-------------|-----------------|
| 0 | Raw Observation | Individual signals: email events, calendar changes, keystroke dynamics, voice features, app usage | ~50 tokens | 100-200/day | Client (ingestion) | voyage-context-3 (1024-dim) |
| 1 | Daily Summary | Narrative summary of a day's observations, anomalies flagged, energy profile, cross-signal co-occurrences | 200-400 tokens | 1/day | Haiku 3.5 | voyage-context-3 (1024-dim) |
| 2 | Behavioural Signature | Recurring patterns validated across time and context — named rules with evidence windows, confidence, cross-domain correlations | 500-1000 tokens | 15-25 validated at month 3, 30-50 at month 6 | Sonnet 4 | voyage-3-large (1024-dim) |
| 3 | Personality Trait | Stable cognitive/behavioural traits with valence vectors, precision scalars, version history, trajectory narratives | 1000-2000 tokens | 3-5 provisional at month 3, 8-12 stable at month 6 | Opus 4.6 | voyage-3-large (1024-dim) |
| ACB | Active Context Buffer | Structured document injected into every LLM call — identity, priorities, relationships, rules, behavioural model | ~2000 tokens | 1 per profile | Opus 4.6 | N/A (not embedded) |

### Conflict Resolution: extract-01 vs extract-03

Extract-01 defines 5 tiers (0-3 + ACB) with specific data structures. Extract-03 defines a 4-stage reflection pipeline (Observation → First-order → Second-order → Traits → Predictions) with its own data structures. These are **compatible, not conflicting**:

- Extract-03's "Tier 0 — Raw Observation" maps directly to extract-01's Tier 0.
- Extract-03's "Tier 1 — Daily Summary" maps directly to extract-01's Tier 1.
- Extract-03's "Tier 2 — Behavioural Signature" maps to extract-01's Tier 2, with extract-03 adding the 4-gate validation protocol (not present in extract-01). **Decision: adopt extract-03's validation gates on extract-01's schema.**
- Extract-03's "Tier 3 — Trait / Executive Model Element" maps to extract-01's Tier 3, but extract-01 adds the valence vector and precision scalar (not in extract-03). **Decision: adopt extract-01's richer trait structure (valence + precision + bi-temporal) with extract-03's versioning and trajectory narrative.**

The prediction layer from extract-03 (Stage 4) is NOT a memory tier — it is an output of the trait model. Predictions live in a separate table (see Section 7).

### Embedding Strategy

| Tiers | Model | Rationale |
|-------|-------|-----------|
| 0, 1 | voyage-context-3 (1024-dim) | Contextual/conversational/event-level memories. Optimised for retrieval where surrounding context matters. |
| 2, 3 | voyage-3-large (1024-dim) | Factual/trait-level memories. Higher accuracy on factual retrieval benchmarks. |

Both models output 1024 dimensions, allowing a single `vector(1024)` column type across all tables. This supersedes the existing Jina v3 decision in the specs.

**Local vector search:** USearch (Swift-native HNSW). Primary query path. Sub-10ms for 500K vectors on Apple Silicon. pgvector is the sync/backup path only.

## 2. Nightly Consolidation Pipeline

### Trigger Conditions

The pipeline runs when any one of these conditions is met:

1. **Time-based:** 2:00 AM local time (primary trigger).
2. **Volume-based:** >200 unprocessed Tier 0 observations.
3. **Significance-based:** Any Tier 0 observation with importance >= 0.9.

### Pipeline Stages

Six phases, executed sequentially. Each phase's output feeds the next. The first three run nightly; Phase 4 runs weekly; Phase 5 runs monthly; Phase 6 (pruning) runs nightly.

This reconciles extract-01's 6-phase pipeline with extract-03's 4-stage reflection model:

| Phase | extract-01 Name | extract-03 Stage | Model | Cadence |
|-------|----------------|-----------------|-------|---------|
| 1 | Importance Scoring | — | Haiku 3.5 | Nightly |
| 2 | Conflict Detection | — | Haiku 3.5 | Nightly |
| 3 | Daily Summary Generation | Stage 1 (Observation → First-order) | Haiku 3.5 | Nightly |
| 4 | Weekly Pattern Detection | Stage 2 (First-order → Second-order) | Sonnet 4 | Weekly |
| 5 | Monthly Trait Reflection | Stages 3+4 (Second-order → Traits → Predictions) | Opus 4.6 | Monthly |
| 6 | Pruning | — | None | Nightly |

Extract-01 adds Phases 1 and 2 (importance scoring, conflict detection) which are absent from extract-03. Extract-03 separates prediction generation into its own Stage 4; here we fold it into Phase 5 (monthly trait reflection) since both require Opus and the predictions depend on the freshly-synthesised trait model.

### Full Pipeline Pseudocode

```
NIGHTLY_CONSOLIDATION_PIPELINE(date):

  // ────────────────────────────────────────────────────────────
  // PHASE 1 — Importance Scoring [Haiku 3.5, nightly]
  // ────────────────────────────────────────────────────────────
  unprocessed = FETCH tier0_observations
    WHERE is_processed = false AND importance_score = 0.5

  FOR EACH obs IN unprocessed:
    score = haiku_classify(
      observation = obs.summary,
      calibration_anchors = IMPORTANCE_ANCHORS,
      system = "Rate this observation's importance on 1-10 using these anchors:
                1 = routine email check, 5 = notable deviation from pattern,
                10 = unprecedented behaviour or critical decision.
                Output only the integer.",
      temperature = 0.0
    )
    UPDATE obs SET importance_score = score / 10.0

  // ────────────────────────────────────────────────────────────
  // PHASE 2 — Conflict Detection [Haiku 3.5, nightly]
  // ────────────────────────────────────────────────────────────
  scored_today = FETCH tier0_observations
    WHERE occurred_at >= date AND is_processed = false

  FOR EACH obs IN scored_today:
    candidates = vector_search(
      query = obs.embedding,
      tiers = [2, 3],
      top_k = 5,
      min_similarity = 0.75
    )
    FOR EACH candidate IN candidates:
      contradiction_score = haiku_assess_contradiction(
        new_observation = obs,
        existing_memory = candidate,
        system = "Does this new observation contradict the existing memory?
                  Score 0.0 (consistent) to 1.0 (directly contradicts).
                  Output only the float.",
        temperature = 0.0
      )
      IF contradiction_score > 0.7:
        TAG obs AS "conflict:{candidate.id}"
        TAG candidate AS "challenged_by:{obs.id}"

  // ────────────────────────────────────────────────────────────
  // PHASE 3 — Daily Summary Generation [Haiku 3.5, nightly]
  // ────────────────────────────────────────────────────────────
  observations_today = FETCH tier0_observations
    WHERE occurred_at >= date ORDER BY occurred_at

  baselines = fetch_current_baselines()

  // Compute baseline deviations for each observation
  FOR EACH obs IN observations_today:
    IF baselines.has(obs.event_type):
      obs.baseline_deviation = compute_sigma_deviation(obs.value, baselines[obs.event_type])

  daily_summary = haiku_generate_summary(
    observations = observations_today,
    baselines = baselines,
    active_context = FETCH active_context_buffer,
    calendar_context = fetch_calendar(date),
    system = "You are a behavioural observation summariser. Describe what happened.
              Do not interpret or explain why. Flag anomalies >1.5σ from baseline.
              Note cross-signal co-occurrences. Structure output as:
              day_narrative, significant_events[], anomalies[], energy_profile.",
    temperature = 0.0
  )

  INSERT INTO tier1_daily_summaries(daily_summary)
  EMBED daily_summary WITH voyage-context-3
  UPDATE tier0_observations SET is_processed = true, processed_at = now()
    WHERE occurred_at >= date

  // ────────────────────────────────────────────────────────────
  // PHASE 4 — Weekly Pattern Detection [Sonnet 4, weekly]
  // ────────────────────────────────────────────────────────────
  IF is_end_of_week(date) OR day_count_this_week >= 5:

    last_7_summaries = FETCH tier1_daily_summaries
      WHERE summary_date >= date - 7 days

    existing_signatures = FETCH tier2_signatures
      WHERE status IN ('emerging', 'developing', 'confirmed')

    candidate_patterns = sonnet_detect_patterns(
      summaries = last_7_summaries,
      existing_signatures = existing_signatures,
      system = "You are a behavioural pattern analyst. Compare this week's daily
                summaries against the existing pattern library. Identify:
                (a) patterns that strengthen existing signatures,
                (b) new candidate patterns appearing for the first time,
                (c) existing patterns that were violated this week.
                Requirements: each pattern must span >=2 domains and >=14 days.
                For each candidate, state the observable rule and the minimum
                evidence needed to validate.",
      temperature = 0.3
    )

    FOR EACH pattern IN candidate_patterns:
      duplicate = vector_search(pattern.embedding, tiers=[2], min_similarity=0.85)

      IF duplicate EXISTS:
        // Reinforce existing signature
        duplicate.last_reinforced = date
        duplicate.observation_count += pattern.new_evidence_count
        duplicate.confidence = recompute_confidence(duplicate)

        // Run 4-gate validation if not yet confirmed
        IF duplicate.status != 'confirmed':
          run_four_gate_validation(duplicate)  // See Section 4
          IF duplicate.passes_all_gates:
            duplicate.status = 'confirmed'
      ELSE:
        // New candidate signature
        INSERT INTO tier2_signatures(
          pattern,
          status = 'emerging',
          first_observed = date,
          confidence = 0.3  // low initial confidence
        )
        EMBED WITH voyage-3-large

  // ────────────────────────────────────────────────────────────
  // PHASE 5 — Monthly Trait Reflection + Prediction [Opus 4.6, monthly]
  // ────────────────────────────────────────────────────────────
  IF is_end_of_month(date) OR month_boundary_crossed:

    // --- STAGE A: Trait Synthesis ---
    confirmed_signatures = FETCH tier2_signatures WHERE status = 'confirmed'
    current_traits = FETCH tier3_traits WHERE valid_to IS NULL
    trait_version_history = FETCH tier3_traits ORDER BY valid_from
    conflict_tagged = FETCH all items tagged "conflict:*" this month

    trait_updates = opus_reflect(
      signatures = confirmed_signatures,
      current_traits = current_traits,
      conflicts = conflict_tagged,
      version_history = trait_version_history,
      system = "You are building the definitive cognitive model of one executive.
                This is your monthly synthesis. Review all validated behavioural
                signatures and the current trait model. For each trait:
                (a) does the evidence still support it? Update precision.
                (b) should any traits be split, merged, or retired?
                (c) do the signatures suggest new traits not yet in the model?
                (d) generate trajectory narrative: how has this person changed?
                Every trait must have a valence vector and precision scalar.
                Handle contradictions using three cases:
                (1) temporary deviation — lower precision, do not update trait
                (2) permanent shift — create new trait version, supersede old
                (3) novel emergence — create new trait with low precision
                Every claim must trace to named signatures.",
      config = { extended_thinking: true, thinking_budget: 65536 },
      temperature = 1.0  // required for extended thinking
    )

    FOR EACH update IN trait_updates:
      CASE update.type:
        temporary_deviation:
          existing_trait.precision *= 0.9
        permanent_shift:
          existing_trait.valid_to = now()
          existing_trait.superseded_by = new_trait.id
          new_trait = INSERT WITH (
            valid_from = now(),
            precision = max(0.5, 1.0 / (1.0 + prediction_error^2)),
            supersedes = existing_trait.id
          )
        novel_emergence:
          INSERT new_trait WITH precision = 0.3
      EMBED all new/updated traits WITH voyage-3-large

    // --- STAGE B: Prediction Generation ---
    current_model = FETCH tier3_traits WHERE valid_to IS NULL AND precision >= 0.7
    upcoming_context = fetch_calendar(next_14_days) + fetch_known_events()

    IF len(current_model) >= 3:  // minimum 3 validated traits
      predictions = opus_generate_predictions(
        model = current_model,
        context = upcoming_context,
        system = "Generate falsifiable predictions. Each MUST specify:
                  (a) the predicted observable behaviour,
                  (b) the time window,
                  (c) the confidence level (0-1),
                  (d) the grounding trait(s),
                  (e) what observation would falsify it.
                  Do not generate predictions where model confidence < 0.6.
                  For cross-trait predictions (requiring 2+ traits), minimum
                  confidence threshold is 0.7.",
        config = { extended_thinking: true, thinking_budget: 65536 },
        temperature = 1.0
      )

      FOR EACH prediction IN predictions:
        IF prediction.confidence >= 0.6:
          INSERT INTO predictions(prediction, status='pending')

    // --- Cross-trait predictions ---
    trait_pairs = generate_pairs(current_model)
    FOR EACH (trait_a, trait_b) IN trait_pairs:
      cross_pred = opus_predict(
        traits = [trait_a, trait_b],
        context = upcoming_context,
        system = "These two traits interact: '{trait_a}' and '{trait_b}'.
                  Given upcoming context, does their interaction predict
                  something neither trait alone would predict?"
      )
      IF cross_pred.confidence >= 0.7:
        INSERT INTO predictions(cross_pred, status='pending')

  // ────────────────────────────────────────────────────────────
  // PHASE 6 — Pruning [nightly]
  // ────────────────────────────────────────────────────────────

  // Tier 0: Cold storage for observations >30 days old
  FOR EACH tier0 WHERE occurred_at < now() - 30 days AND importance_score < 0.8:
    MOVE raw_data to Supabase Storage bucket
    REPLACE raw_data with storage URI reference

  // Tier 0: Tombstone observations >365 days old
  FOR EACH tier0 WHERE occurred_at < now() - 365 days
    AND importance_score < 0.8 AND is_processed = true:
    TOMBSTONE (retain summary + embedding, purge raw_data reference)

  // Tier 2: Fade stale signatures
  FOR EACH tier2 WHERE last_reinforced < now() - 60 days AND status = 'confirmed':
    UPDATE status = 'fading'

  // Tier 3: NEVER prune. Superseded traits remain with valid_to set.

  // Update active context buffer
  UPDATE active_context_buffer WITH (
    latest Tier 3 traits (valid_to IS NULL),
    top Tier 2 signatures (status = 'confirmed', ORDER BY confidence DESC LIMIT 8),
    current session context
  )
```

## 3. Retrieval System

### 5-Dimension Composite Scoring

Every memory retrieval query is scored across 5 dimensions, weighted by the query's intent type.

```
composite_score(memory, query, intent) =
    w_recency(intent)    * recency_decay(memory)
  + w_importance(intent) * importance(memory)
  + w_relevance(intent)  * cosine_similarity(memory.embedding, query.embedding)
  + w_temporal(intent)   * temporal_match(memory, query.time_range)
  + w_tier_boost(intent) * tier_weight(memory.tier)
```

**Dimension definitions:**

| Dimension | Formula | Notes |
|-----------|---------|-------|
| Recency | `exp(-0.005 * hours_since(occurred_at))` | Half-life ~139 hours (~5.8 days). Ensures recent events surface but old high-tier memories aren't buried. |
| Importance | Direct from `importance_score` (Tier 0) or `confidence` (Tier 2) or `precision` (Tier 3) | Tier 1 uses fixed 0.6. |
| Relevance | `1 - (embedding <=> query_embedding)` | Cosine similarity via pgvector or USearch. |
| Temporal | 1.0 if memory falls within query's time range, 0.3 otherwise | Rewards memories from the asked-about period. |
| Tier Boost | Tier 0: 0.2, Tier 1: 0.4, Tier 2: 0.7, Tier 3: 1.0 | Higher tiers are more valuable abstractions. |

### Intent-Aware Weight Table

Queries are classified into 5 intent types. Each triggers a different weight profile. Classification is done by the calling LLM (Haiku for simple routing, Sonnet for ambiguous queries).

| Intent | w_recency | w_importance | w_relevance | w_temporal | w_tier_boost | Use Case |
|--------|-----------|-------------|-------------|------------|-------------|----------|
| WHAT | 0.15 | 0.20 | 0.50 | 0.05 | 0.10 | "What is her communication style?" — relevance-dominated |
| WHEN | 0.10 | 0.15 | 0.25 | 0.40 | 0.10 | "When did she last defer a 1:1?" — temporal-dominated |
| WHY | 0.10 | 0.25 | 0.40 | 0.10 | 0.15 | "Why did she reverse the hiring decision?" — relevance + importance |
| PATTERN | 0.05 | 0.30 | 0.30 | 0.10 | 0.25 | "What patterns emerge around board meetings?" — tier + importance |
| CHANGE | 0.20 | 0.20 | 0.30 | 0.20 | 0.10 | "How has her risk appetite changed?" — balanced, recency matters |

### Hierarchical Drill-Down Retrieval

For complex queries, retrieval proceeds top-down through the hierarchy:

```
FUNCTION hierarchical_retrieve(query, intent):
  // Phase 1: Search traits first (Tier 3)
  trait_matches = vector_search(query, tiers=[3], top_k=5)

  // Phase 2: For each matching trait, retrieve supporting signatures (Tier 2)
  signature_matches = []
  FOR EACH trait IN trait_matches:
    sigs = FETCH tier2_signatures WHERE id IN trait.evidence_chain.supporting_tier2_ids
    signature_matches.append(sigs)

  // Phase 3: If more detail needed, drill into daily summaries (Tier 1)
  IF intent IN ('WHY', 'CHANGE'):
    FOR EACH sig IN signature_matches:
      summaries = FETCH tier1_daily_summaries
        WHERE id IN sig.supporting_tier1_ids

  // Phase 4: Only drill to raw observations for evidence chains
  IF intent == 'WHY':
    FOR EACH summary IN summaries:
      observations = FETCH tier0_observations
        WHERE id IN summary.source_tier0_ids
        ORDER BY importance_score DESC LIMIT 5

  RETURN ranked_union(trait_matches, signature_matches, summaries, observations)
```

### Retrieval Floor

Any memory with composite_score < 0.15 is excluded. If no memories exceed this threshold, return empty with a `low_confidence` flag rather than surfacing low-quality results.

| Table | Index Type | Parameters | Purpose |
|-------|-----------|------------|---------|
| tier0_observations | HNSW | m=16, ef_construction=128 | High-volume vector search, lower recall acceptable |
| tier0_observations | BRIN | on occurred_at | Efficient temporal range scans on append-only data |

## 4. Compounding Mechanism

### Phase Transitions

Intelligence compounds **structurally, not volumetrically**. Month-6 intelligence is impossible at month 1 because it depends on representational layers that cannot exist until earlier layers have been built and validated.

| Period | Phase | What Becomes Possible | Why Impossible Before | Memory State |
|--------|-------|----------------------|----------------------|-------------|
| Week 1-2 | Baseline establishment | Raw behavioural fingerprint — email cadence, calendar structure, keystroke baseline, app usage norms | Nothing to compare against; all observations are first-instance | Tier 0: ~2500 obs, Tier 1: 7-14 summaries, Tier 2: 0 (no pattern promotion before week 3), Tier 3: 0 |
| Month 1 | Routine detection | Recurring daily/weekly rhythms, sender-response priority map, energy cycle approximation | Requires minimum 3-4 repetitions of weekly cycles to distinguish signal from noise | Tier 0: ~10K obs, Tier 1: ~30 summaries, Tier 2: 5-10 patterns (Gates 1+2 passed), Tier 3: 0 |
| Month 3 | Deviation detection | Anomalies flagged against stable baselines, first cross-signal correlations (calendar compression + email latency = stress proxy), first low-confidence predictions | Baselines must exist before deviations detected; cross-signal requires confirmed independent patterns first | Tier 0: ~30K (pruned to 30-day window + flagged), Tier 1: ~90, Tier 2: 15-25 validated (all 4 gates), Tier 3: 3-5 provisional |
| Month 6 | Predictive capability | Forward-looking predictions. Trait-level model stable enough for novel-situation reasoning. Burnout prediction entering useful range (Brier < 0.3). | Prediction requires validated trait models (month 3+), plus enough historical predictions to calibrate confidence | Tier 0: ~30K rolling, Tier 1: ~180, Tier 2: 30-50 validated, Tier 3: 8-12 stable |
| Month 12 | Deep model / anticipatory | Anticipates reactions to novel situations. Detects multi-month trajectories. Comparable to trusted Chief of Staff after 50 sessions. | Trajectory detection requires 6+ months of versioned trait snapshots. Novel-situation anticipation requires stable, validated cognitive model | Tier 0: ~30K rolling, Tier 1: ~365, Tier 2: 50+ validated, Tier 3: 12-20 with full version histories |

### Pattern Validation — Four-Gate Protocol

Every candidate pattern at Tier 2 must pass all four gates before promotion to `confirmed` status. This is a hard architectural constraint — there is no override.

```
FOUR_GATE_VALIDATION(signature):

  // GATE 1: Statistical Significance
  // ARIMA-corrected Cohen's d_z >= 0.5 (medium within-person effect)
  // ARIMA correction removes autocorrelation from time series before computing
  // effect size — raw Cohen's d on autocorrelated behavioural data inflates significance.
  time_series = fetch_signal_timeseries(
    signal_type = signature.signal_type,
    window = signature.evidence_window
  )
  arima_residuals = fit_arima(time_series).residuals
  baseline_residuals = arima_residuals[baseline_period]
  pattern_residuals = arima_residuals[pattern_period]
  cohens_dz = mean(pattern_residuals - baseline_residuals) /
              std(pattern_residuals - baseline_residuals)
  gate_1_pass = abs(cohens_dz) >= 0.5

  // GATE 2: Temporal Stability
  // Test-retest correlation across non-adjacent weeks >= 0.6
  // Pattern must hold in at least 3 of 4 sampled weeks.
  // HARD CONSTRAINT: Cannot run until at least 3 non-adjacent weeks exist.
  weekly_values = partition_by_week(time_series)
  IF len(weekly_values) < 3:
    gate_2_pass = false  // insufficient data, cannot evaluate
  ELSE:
    sample_pairs = select_non_adjacent_week_pairs(weekly_values, n_pairs=4)
    test_retest_r = mean([pearson_r(pair.week_a, pair.week_b) for pair in sample_pairs])
    gate_2_pass = test_retest_r >= 0.6

  // GATE 3: Contextual Replication
  // Pattern replicates across at least 2 distinct context conditions.
  // If a pattern only appears in one context, it's a contextual response, not a trait.
  contexts = partition_by_context(
    time_series,
    context_variables = ["calendar_load", "day_of_week", "meeting_density"]
  )
  context_results = []
  FOR EACH context_group IN contexts:
    IF len(context_group) >= 10:  // minimum sample per context
      group_dz = compute_cohens_dz(context_group)
      context_results.append(abs(group_dz) >= 0.3)  // relaxed threshold per-context
  gate_3_pass = sum(context_results) >= 2 AND len(context_results) >= 2

  // GATE 4: Causal Coherence
  // LLM review — the pattern must map to a known cognitive/behavioural mechanism.
  // Pure statistical artifacts with no plausible mechanism are quarantined.
  coherence = sonnet_evaluate(
    system = "This statistical pattern was detected: {signature.description}.
              Effect size: {cohens_dz}. Does this map to a known cognitive,
              behavioural, or organisational mechanism? If yes, name it and
              explain the mechanism in 1-2 sentences. If no plausible mechanism
              exists, flag as potential artifact.",
    temperature = 0.3,
    max_tokens = 500
  )
  gate_4_pass = coherence.mechanism_identified == true

  // DECISION
  signature.validation_gates = {gate_1_pass, gate_2_pass, gate_3_pass, gate_4_pass}
  IF all gates pass:
    signature.status = 'confirmed'
    signature.confidence = compute_weighted_confidence(
      cohens_dz, test_retest_r, context_results, coherence
    )
  ELSE:
    signature.status remains current (stays 'emerging' or 'developing')
    signature.next_revalidation = now() + 7 days
```

### Statistical Thresholds (Complete Reference)

| Metric | Threshold | Meaning |
|--------|-----------|---------|
| Cohen's d_z (Gate 1) | >= 0.5 | Medium within-person effect |
| Test-retest r (Gate 2) | >= 0.6 | Moderate temporal stability |
| Context replication (Gate 3) | >= 2 contexts with d_z >= 0.3 | Pattern is cross-contextual |
| Causal coherence (Gate 4) | Mechanism identified by Sonnet | Psychologically interpretable |
| Pattern promotion minimum | Week 3 | No promotion before 3 non-adjacent weeks exist |
| BOCPD change point probability | >= 0.7 | Sufficient evidence to trigger quarantine |
| CDI quarantine confirmation | >= 0.65 | Change is genuine, not transient |
| Prediction surfacing minimum | confidence >= 0.6 | Below this, prediction is never shown |
| Cross-trait prediction minimum | confidence >= 0.7 | Higher bar for compounded uncertainty |
| Trait precision for prediction | >= 0.7 | Traits below this cannot generate predictions |

### Change Detection — BOCPD + Quarantine

Bayesian Online Change Point Detection (Adams & MacKay, 2007) runs simultaneously at three abstraction levels:

1. **Signal level:** Raw metric change points (email response time suddenly shifts).
2. **Pattern level:** Behavioural rule violations (established pattern stops holding).
3. **Trait level:** Executive model predictions systematically failing.

```
BOCPD_MONITOR(signal_stream):
  levels = [SIGNAL, PATTERN, TRAIT]

  FOR EACH level IN levels:
    run_length_distribution = initialize_uniform()
    hazard_rate = 1/250  // expect a change every ~250 observations

    FOR EACH new_observation IN signal_stream[level]:
      // Compute predictive probability under each run length
      FOR EACH r IN run_length_distribution:
        predictive_prob[r] = student_t_predictive(
          observation = new_observation,
          sufficient_stats = stats[r]
        )

      // Growth probability (no change point)
      growth = run_length_distribution * predictive_prob * (1 - hazard_rate)

      // Change point probability
      changepoint = sum(run_length_distribution * predictive_prob * hazard_rate)

      // Update distribution
      run_length_distribution = normalize(append(changepoint, growth))
      update_sufficient_stats(new_observation)

      // Detection threshold
      IF changepoint_probability > 0.7:
        trigger_quarantine(level, new_observation.timestamp)


QUARANTINE_PROTOCOL(level, change_timestamp):
  quarantine_window = 14 days
  pre_change = fetch(change_timestamp - 30d, change_timestamp)
  post_change = fetch(change_timestamp, change_timestamp + 14d)

  // Composite Deviation Index
  CDI = (
    0.40 * magnitude_of_shift(pre_change, post_change) +
    0.35 * consistency_in_quarantine(post_change) +
    0.25 * cross_level_concordance(levels_flagging_change)
  )

  IF CDI >= 0.65:
    // Genuine change — version the trait
    old_trait = fetch_current_trait(level)
    old_trait.valid_to = change_timestamp
    new_trait = synthesize_new_trait(post_change, old_trait)
    new_trait.valid_from = change_timestamp
    new_trait.precision = 0.5  // uncertain about new state
    store(new_trait)
    generate_trajectory_narrative(old_trait, new_trait)
  ELSE:
    // Transient perturbation — log as episodic, do NOT update trait model
    store_episodic_event(change_timestamp, "transient_deviation", post_change)
```

### Cathartic Update Formula (Precision Scalar)

When a contradiction is detected against a Tier 3 trait:

```
prediction_error = |observed_value - trait.valence_vector[dimension]|

IF prediction_error < 0.3:
  // Noise — within expected variance
  trait.precision *= 0.95  // slight precision reduction

ELSE IF prediction_error >= 0.3 AND prediction_error < 0.7:
  // Significant deviation — add to contradiction log, quarantine
  ADD to trait.contradiction_log
  IF contradiction_count_in_14_days >= 3:
    // Permanent shift confirmed
    new_precision = 1.0 / (1.0 + prediction_error^2)
    CREATE new trait version with updated valence, precision = new_precision
    SUPERSEDE old trait
  // Else: wait for more evidence

ELSE:  // prediction_error >= 0.7
  // Dramatic shift — immediate update
  CREATE new trait version
  old_trait.valid_to = now()
  new_trait.precision = 0.3  // very low — system is uncertain
```

This formula interacts with BOCPD: if BOCPD detects a change point and the cathartic update independently flags a dramatic shift, the two mechanisms converge to produce a high-confidence trait version change. If only one triggers, the quarantine period provides a safety buffer.

## 5. Behavioural Prediction Layer

The prediction layer sits above the memory tiers. It reads from Tiers 2 and 3 to generate forward-looking predictions. It writes to dedicated prediction tables (not to memory tiers). It is the primary mechanism by which accumulated intelligence produces actionable output.

### 5.1 Avoidance Detection

**Signal model:** Three-stream detection, each mapping to a distinct avoidance mechanism:

| Stream | Signal | Psychological Construct | Threshold |
|--------|--------|------------------------|-----------|
| Email latency | Sender-specific response time z-score | Experiential avoidance (Hayes) | z > 2.0 for specific sender cluster, persists > 5 business days |
| Calendar rescheduling | Meeting-type-specific reschedule rate | Approach-avoidance conflict (Lewin/Miller) | Reschedule rate > 2σ above baseline for 2+ consecutive weeks |
| Document engagement | Topic-specific open-without-edit events | Cognitive avoidance (Borkovec) | Document opened 3+ times without meaningful edit (< 5% change) within decision deadline |

**Cross-stream fusion:** Avoidance is declared when >= 2 of 3 streams flag simultaneously for the same decision domain. Domain is inferred from sender/meeting/document topic clustering.

**Strategic delay vs. avoidance discriminator:** Information network analysis. If the executive's information network is expanding (new sources consulted, new stakeholders contacted), classify as strategic delay. If the network is static or contracting (re-processing same material, no new consultations), confirm avoidance.

**Hard rule:** NEVER flag avoidance if information network is expanding. Default to "strategic delay" when ambiguous. False negatives are cheaper than false positives — wrongly accusing an executive of avoiding destroys trust.

**Integration with memory tiers:** Avoidance patterns, once confirmed across >= 3 occurrences, are promoted to Tier 2 behavioural signatures. If the avoidance pattern is domain-specific and stable, it may eventually contribute to a Tier 3 trait (e.g., "Confrontation-Avoidant Decision Maker").

### 5.2 Burnout Prediction Pipeline

**Signal taxonomy:** 17 behavioural signals mapped to the three Maslach Burnout Inventory (MBI) dimensions:

| MBI Dimension | Digital Signal Correlates |
|---------------|-------------------------|
| Emotional exhaustion | After-hours email ratio, voice F0 range compression, keystroke error rate |
| Depersonalisation | Communication network contraction, email brevity increase, meeting attendance decline |
| Reduced personal accomplishment | Calendar buffer erosion, increased time-to-decision, document open-without-edit patterns |

**Leading indicator temporal order:** After-hours email ratio increase → response time distribution widening → calendar buffer erosion → voice prosody flattening → keystroke dynamics degradation → communication network contraction.

**Model architecture:** LSTM + XGBoost ensemble. LSTM captures temporal trajectory shape (stable/deteriorating/recovering/volatile). XGBoost handles cross-signal feature interactions at each time step. Ensemble weighting learned from validation set.

**Triple gate for false positive control:**

1. **Duration gate:** Deteriorating trajectory sustained >= 3 weeks. No alert on transient spikes.
2. **Stream consistency gate:** Signals from >= 3 different modalities must converge.
3. **MBI convergence gate:** 2-of-3 MBI dimensions must show elevation (emotional exhaustion + depersonalisation, OR emotional exhaustion + reduced accomplishment).

**Sprint discriminator:** If a workload spike has a visible end date within 4 weeks on the calendar AND communication network breadth is stable, suppress burnout scoring for that period. Sprints are intense but not burnout.

**Key numbers:**
- Earliest reliable prediction: 8-12 weeks of observation
- Lead time before self-report: 4-8 weeks (intervention window)
- False positive rate without gating: 20-35%. With triple gate: estimated 8-12%.
- Minimum observation: 8 weeks baseline + 3 weeks sustained signal = 11 weeks before first burnout alert.

### 5.3 Decision Reversal Model

**Architecture:** Cox proportional hazards survival model for time-to-reversal probability, combined with a 4-state Hidden Markov Model for post-decision state tracking.

**HMM states:** Committed → Consolidating → Wavering → Reversing

**Key features:**

| Feature | Signal | Source |
|---------|--------|--------|
| Time-to-initial-decision | Fast decisions reversed more often | Calendar + email timestamps |
| Post-decision info seeking | Volume of new searches/docs on topic | App monitoring |
| New information sources | Previously unconsulted sources entering picture | Network analysis |
| Same-source revisits | Re-processing indicator | Document tracking |
| Stakeholder re-explanations | Mentions/discussions of the decision to stakeholders | Email/calendar |
| Consultation breadth | Narrow consultation → higher reversal risk | Meeting + email analysis |

**Temporal horizon:** Strongest predictive signal in T+4 to T+14 day window after initial decision. Before T+4, insufficient post-decision behaviour. After T+14, most decisions have consolidated or already reversed.

**Healthy reconsideration vs. problematic vacillation:** Discriminated via network analysis — new external information sources = healthy reconsideration. Re-processing same material with no new inputs = vacillation. Grounded in Svenson's Differentiation and Consolidation (DiffCon) theory.

### 5.4 Bayesian Individual Decision Model

**Architecture:** Hierarchical Bayesian model (Griffiths & Tenenbaum framework).

- Population-level priors from executive decision-making literature (Upper Echelons theory — Hambrick).
- Individual-level parameters updated per observed decision.
- Population prior dominated by individual data after ~5 observations per domain.

**Granularity:** Per-decision-domain models (financial, personnel, strategic, operational). NOT one unified model. Decision style varies by domain.

**Context features:** Stakeholders involved (count + power dynamics), financial magnitude (log-scaled), time pressure (deadline proximity), domain familiarity (inferred from historical frequency), reversibility of the decision.

**Concept drift handling:** Exponential forgetting factor on older observations. BOCPD (Adams & MacKay, 2007) runs in parallel — when changepoint probability > 0.8, reset domain prior to weakly informative and begin relearning. This threshold is higher than the memory-tier BOCPD (0.7) because domain model reset is a more aggressive action.

**Prediction suppression:** Effective sample size (ESS) tracked per domain. ESS < 10 → suppress predictions for that domain entirely. Report model uncertainty instead.

### 5.5 Temporal Sequence Modelling

**Primary architecture:** Temporal Fusion Transformer (TFT) — Lim et al., 2021.

Selection rationale:
1. Native multi-horizon forecasting
2. Built-in variable selection networks handle heterogeneous inputs (email timing, calendar structure, keystroke dynamics, voice features) without manual feature engineering
3. Interpretable attention weights — essential for executive-facing explanations
4. Handles irregularly-sampled time series natively

**Bayesian head:** Replace TFT's quantile output layer with a Bayesian probabilistic head for calibrated uncertainty estimates. This gives the hybrid TFT-Bayesian architecture — neural feature extraction + probabilistic prediction.

**Foundation model integration (Phase 2):** Chronos (Amazon, 2024) and MOIRAI (Salesforce, 2024) used as zero-shot anomaly detectors in parallel with TFT. At month 12+ with sufficient individual data, LoRA fine-tuning on accumulated data for personalised forecasting.

**Mamba/S4 state-space models:** Reserved for Phase 2 comparison when observation windows exceed 12+ months of high-frequency data and O(n) complexity advantage over TFT's O(n^2) attention becomes relevant.

### 5.6 Prediction Confidence & Surfacing

**Phase-gated trust-building sequence:**

| Period | What Surfaces | Confidence Threshold | Rationale |
|--------|-------------|---------------------|-----------|
| Months 1-3 | Descriptive only — "here's what happened" summaries, no predictions | N/A | System learns baselines. Has not earned the right to predict. |
| Months 4-6 | Low-stakes predictions — meeting duration, response time, workload forecasts | P >= 0.70 | Build track record on predictions that can't damage trust. |
| Month 7+ | High-stakes predictions — burnout trajectory, decision reversal risk, avoidance detection | P >= 0.85-0.90, ESS >= 10 | Only after 6 months of demonstrated accuracy. |

**Self-defeating prophecy resolution:** A prediction that triggers behaviour change is a *successful intervention*, not a false positive. Track prediction-triggered behaviour changes as positive outcomes. Calibration metrics must account for this.

**UX language:** Never "you will burn out." Always "patterns consistent with increasing load — here's what's driving it." Never certainties. Always "patterns observed" not "you will." Give the executive control to dismiss predictions — the system learns from dismissals.

**Error recovery:** After a visible error, the system acknowledges it explicitly ("last week's pattern didn't play out — here's what was different") rather than silently moving on. Transparent error acknowledgment rebuilds trust faster.

### 5.7 Integration with Memory Tiers

The prediction layer is a consumer of the memory system, not part of it:

```
MEMORY TIERS (persistent store)        PREDICTION LAYER (derived outputs)
┌──────────────────────┐               ┌───────────────────────────┐
│ Tier 0: Observations │──reads──────> │ Signal-level BOCPD        │
│ Tier 1: Summaries    │──reads──────> │ Burnout signal ingestion  │
│ Tier 2: Signatures   │──reads──────> │ Pattern-level BOCPD       │
│                      │──reads──────> │ Avoidance detection       │
│ Tier 3: Traits       │──reads──────> │ Decision model            │

│ ACB: Context Buffer  │──reads──────> │ All prediction contexts   │
└──────────────────────┘               └───────────────────────────┘
                                                    │
```

## 6. Evaluation Framework

### Compounding Contribution Ratio (CCR)

The key empirical test for whether the system is genuinely compounding or merely accumulating data.

```
CCR = accuracy_of_model_predictions / accuracy_of_baseline_predictions
```

Where:
- **Model predictions** require the accumulated model (trait-based, cross-signal, trajectory-aware).
- **Baseline predictions** are achievable from raw data alone (base-rate, last-7-days heuristic).

| CCR Value | Interpretation |
|-----------|---------------|
| CCR <= 1.0 | Compounding mechanism is broken. The accumulated model adds no value beyond fresh data. |
| CCR 1.0-1.5 | Marginal compounding. Model adds some value but not yet transformative. |
| CCR > 1.5 | Strong compounding. Model produces insights structurally impossible from raw data. |

**Expected trajectory:**

| Period | CCR | Brier (Behavioural) | Brier (Strategic) |
|--------|-----|--------------------|--------------------|
| Month 3 | 1.0-1.1 | 0.35-0.40 | — |
| Month 6 | 1.3-1.5 | 0.25-0.30 | 0.40-0.45 |
| Month 9 | 1.7-2.0 | 0.20-0.25 | 0.30-0.35 |
| Month 12 | 2.0-2.5 | 0.15-0.20 | 0.25-0.30 |

### Brier Score Tracking

Every falsifiable prediction gets a Brier score: `Brier = (1/N) * Σ(predicted_probability - outcome)^2`. Range 0-1, lower is better. Track monthly rolling average. The system is compounding if Brier scores decrease monotonically over months.

### Counterfactual Testing

Monthly: run a "fresh system" simulation — give a fresh Opus instance access to raw data (Tier 0 + Tier 1 of the current period) but no accumulated model (no Tier 2, no Tier 3, no version history). Compare its insight quality to the production system's. If the fresh system matches the production system, the model is not compounding.

Four conditions that together constitute proof of compounding:
1. Brier scores declining month-over-month
2. CCR > 1.0 and increasing
3. Counterfactual gap widening (production system increasingly outperforms fresh system)
4. Insight tier distribution shifting upward (more trait-level and prediction-level outputs, fewer observation-level)

### Prediction Accuracy Tracking

Every prediction stored with:
- `prediction_resolved: Bool` — has the time window elapsed?
- `actual_outcome: String` — what actually happened
- `brier_score: Float` — calibration error
- `behaviour_change_detected: Bool` — did the executive act on the prediction? (self-defeating prophecy = success)
- `calibration_error: Float` — `|predicted_probability - actual_outcome_binary|`

### Implicit Behavioural Validation

Do NOT use surveys or explicit ratings ("Was this insight useful?"). This creates confirmation bias and survey fatigue. Instead track:
- Dwell time on briefing sections
- Actions taken after recommendations
- Which briefing sections are opened vs. skipped
- Whether the executive references insights in their own communications

### Anti-Pattern: Accumulation vs. Compounding

**Accumulation** = more data → more examples of same patterns → marginally higher confidence. A system with 100K observations producing the same tier of insight as one with 30K is accumulating.

**Genuine compounding** = more data → new representational layers → qualitatively different intelligence. The test: can the system produce insights at month 6 that are structurally impossible at month 3, regardless of data volume?

Failure modes that produce accumulation instead of compounding:
- Flat memory: all observations at same tier, no abstraction hierarchy
- Missing Stage 3: patterns detected but never synthesised into traits
- Missing Stage 4: traits exist but never used for prediction
- No version history: model updates overwrite history, destroying trajectory detection
- No cross-signal correlation: each signal analysed independently

```sql
-- RETRIEVAL FUNCTION

CREATE OR REPLACE FUNCTION retrieve_memories(
    p_profile_id UUID,
    p_query_embedding vector(1024),
    p_intent_type TEXT,           -- 'WHAT', 'WHEN', 'WHY', 'PATTERN', 'CHANGE'
    p_tiers INT[],                -- which tiers to search: {0,1,2,3}
    p_top_k INT DEFAULT 25,
    p_min_importance REAL DEFAULT 0.0,
    p_temporal_start TIMESTAMPTZ DEFAULT NULL,
    p_temporal_end TIMESTAMPTZ DEFAULT NULL
) RETURNS TABLE (
    memory_id UUID,
    tier INT,
    content TEXT,
    importance REAL,
    occurred_at TIMESTAMPTZ,
    cosine_similarity REAL,
    composite_score REAL
) AS $$
DECLARE
    w_recency REAL;
    w_importance REAL;
    w_relevance REAL;
    w_temporal REAL;
    w_tier_boost REAL;
BEGIN
    -- Intent-aware weight selection
    CASE p_intent_type
        WHEN 'WHAT' THEN
            w_recency := 0.15; w_importance := 0.20; w_relevance := 0.50;
            w_temporal := 0.05; w_tier_boost := 0.10;
        WHEN 'WHEN' THEN
            w_recency := 0.10; w_importance := 0.15; w_relevance := 0.25;
            w_temporal := 0.40; w_tier_boost := 0.10;
        WHEN 'WHY' THEN
            w_recency := 0.10; w_importance := 0.25; w_relevance := 0.40;
            w_temporal := 0.10; w_tier_boost := 0.15;
        WHEN 'PATTERN' THEN
            w_recency := 0.05; w_importance := 0.30; w_relevance := 0.30;
            w_temporal := 0.10; w_tier_boost := 0.25;
        WHEN 'CHANGE' THEN
            w_recency := 0.20; w_importance := 0.20; w_relevance := 0.30;
            w_temporal := 0.20; w_tier_boost := 0.10;
        ELSE
            w_recency := 0.20; w_importance := 0.25; w_relevance := 0.35;
            w_temporal := 0.10; w_tier_boost := 0.10;
    END CASE;

    RETURN QUERY
    WITH candidates AS (
        -- Tier 0
        SELECT t0.id AS memory_id, 0 AS tier, t0.summary AS content,
               t0.importance_score AS importance, t0.occurred_at,
               1 - (t0.embedding <=> p_query_embedding) AS cosine_similarity
        FROM tier0_observations t0
        WHERE 0 = ANY(p_tiers)
          AND t0.profile_id = p_profile_id
          AND t0.importance_score >= p_min_importance
          AND t0.embedding IS NOT NULL
          AND (p_temporal_start IS NULL OR t0.occurred_at >= p_temporal_start)
          AND (p_temporal_end IS NULL OR t0.occurred_at <= p_temporal_end)
        ORDER BY t0.embedding <=> p_query_embedding LIMIT p_top_k * 2

        UNION ALL

        -- Tier 1
        SELECT t1.id, 1, t1.day_narrative, 0.6, t1.summary_date::TIMESTAMPTZ,
               1 - (t1.embedding <=> p_query_embedding)
        FROM tier1_daily_summaries t1
        WHERE 1 = ANY(p_tiers)
          AND t1.profile_id = p_profile_id
          AND t1.embedding IS NOT NULL
          AND (p_temporal_start IS NULL OR t1.summary_date >= p_temporal_start::DATE)
          AND (p_temporal_end IS NULL OR t1.summary_date <= p_temporal_end::DATE)
        ORDER BY t1.embedding <=> p_query_embedding LIMIT p_top_k

        UNION ALL

        -- Tier 2
        SELECT t2.id, 2, t2.description, t2.confidence, t2.period_start::TIMESTAMPTZ,
               1 - (t2.embedding <=> p_query_embedding)
        FROM tier2_signatures t2
        WHERE 2 = ANY(p_tiers)
          AND t2.profile_id = p_profile_id
          AND t2.status IN ('developing', 'confirmed')
          AND t2.embedding IS NOT NULL
        ORDER BY t2.embedding <=> p_query_embedding LIMIT p_top_k

        UNION ALL

        -- Tier 3
        SELECT t3.id, 3, t3.description, t3.precision, t3.valid_from,
               1 - (t3.embedding <=> p_query_embedding)
        FROM tier3_traits t3
        WHERE 3 = ANY(p_tiers)
          AND t3.profile_id = p_profile_id
          AND t3.valid_to IS NULL
          AND t3.embedding IS NOT NULL
        ORDER BY t3.embedding <=> p_query_embedding LIMIT p_top_k
    )
    SELECT c.memory_id, c.tier, c.content, c.importance, c.occurred_at,
           c.cosine_similarity,
           (w_recency * exp(-0.005 * EXTRACT(EPOCH FROM (now() - c.occurred_at)) / 3600.0))
           + (w_importance * c.importance)
           + (w_relevance * c.cosine_similarity)
           + (w_temporal * CASE
               WHEN p_temporal_start IS NOT NULL
                    AND c.occurred_at BETWEEN p_temporal_start AND COALESCE(p_temporal_end, now())
               THEN 1.0 ELSE 0.3 END)
           + (w_tier_boost * CASE c.tier
               WHEN 3 THEN 1.0
               WHEN 2 THEN 0.7
               WHEN 1 THEN 0.4
               WHEN 0 THEN 0.2
               END)
           AS composite_score
    FROM candidates c
    WHERE (w_recency * exp(-0.005 * EXTRACT(EPOCH FROM (now() - c.occurred_at)) / 3600.0))
           + (w_importance * c.importance)
           + (w_relevance * c.cosine_similarity)
           + (w_temporal * CASE
               WHEN p_temporal_start IS NOT NULL
                    AND c.occurred_at BETWEEN p_temporal_start AND COALESCE(p_temporal_end, now())
               THEN 1.0 ELSE 0.3 END)
           + (w_tier_boost * CASE c.tier
               WHEN 3 THEN 1.0 WHEN 2 THEN 0.7 WHEN 1 THEN 0.4 WHEN 0 THEN 0.2 END)
           >= 0.15  -- retrieval floor
    ORDER BY composite_score DESC
    LIMIT p_top_k;
END;
$$ LANGUAGE plpgsql STABLE;
```

## Appendix B: Conflict Resolution Log

Decisions made where the three source documents conflicted or were ambiguous:

1. **Tier structure (extract-01: 5-tier vs extract-03: 4-stage):** Adopted extract-01's 5-tier storage model (Tiers 0-3 + ACB) as the physical architecture, with extract-03's 4-stage pipeline as the processing model that populates those tiers. The "predictions" from extract-03's Stage 4 are outputs, not a memory tier — they live in a separate `predictions` table.

2. **Tier 2 validation (extract-01: confidence only vs extract-03: 4-gate protocol):** Adopted extract-03's 4-gate validation protocol as the mechanism that determines confidence. Extract-01's Tier 2 had confidence but no defined validation method. The gates now produce the confidence score.

3. **Tier 3 structure (extract-01: valence + precision vs extract-03: version + trajectory):** Merged both. Extract-01's valence vector + precision scalar + bi-temporal model is the base structure. Extract-03's version number, predictions_enabled, and trajectory_narrative are added fields.
