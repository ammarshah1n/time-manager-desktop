# PRD: Procedural Memory Store

> Layer: Memory Store (Tier 3 of 3)
> Status: Implementation-ready
> Dependencies: Semantic Memory, Episodic Memory, Reflection Engine, PlanningEngine
> Swift target: 5.9+ / macOS 14+
> Reference: Park et al. 2023 (Generative Agents), existing BehaviourRule in PlanningEngine.swift

---

## 1. Purpose

Procedural memory stores the operating rules that Timed generates from its understanding of the executive. If episodic memory is "what happened" and semantic memory is "what we know about this person," procedural memory is "what we should DO with that knowledge."

These rules are the executable output of the intelligence system. They constrain scheduling, shape morning briefings, trigger alerts, and guide every interaction. They are generated by the reflection engine, tested against reality, tracked for effectiveness, and deprecated when they stop working.

Procedural memory is what makes Timed act intelligently — not through general-purpose AI reasoning on every interaction, but through accumulated, validated, specific rules about THIS executive.

## 2. Schema

### 2.1 Procedural Rule Record

```swift
struct ProceduralRule: Identifiable, Codable, Sendable {
    let id: UUID
    let workspaceId: UUID
    let profileId: UUID
    let ruleText: String                      // natural language: "Schedule HR tasks for Tuesday mornings"
    let triggerConditions: [TriggerCondition]  // structured conditions for programmatic evaluation
    let action: RuleAction                    // what the rule prescribes
    let confidence: Float                     // 0.0 to 1.0
    let evidenceChain: [UUID]                 // episodic episode IDs that led to this rule
    let sourceFactIds: [UUID]                 // semantic fact IDs this rule is derived from
    let evidenceCount: Int                    // total supporting episodes
    let activationCount: Int                  // how many times this rule has been applied
    let successCount: Int                     // how many activations led to positive outcomes
    let failureCount: Int                     // how many activations led to negative outcomes
    let lastActivated: Date?                  // when the rule was last applied
    let exceptions: [RuleException]           // known conditions where this rule should NOT fire
    let status: RuleStatus
    let proposedAt: Date                      // when the reflection engine first generated this rule
    let activatedAt: Date?                    // when it moved from proposed to active
    let deprecatedAt: Date?                   // when it was deprecated
    let deprecationReason: String?            // why it was deprecated
    let createdAt: Date
    let updatedAt: Date
}
```

### 2.2 Trigger Conditions

```swift
struct TriggerCondition: Codable, Sendable {
    let field: TriggerField           // what to evaluate
    let op: TriggerOperator           // how to compare
    let value: String                 // comparison value (JSON-encoded for complex types)
}

enum TriggerField: String, Codable, Sendable {
    // Task properties
    case taskBucket               // "Reply", "Action", etc.
    case taskCategory             // derived from title/sender: "HR", "Finance", "Client"
    case taskSender               // email sender domain or name
    case taskPriority             // 1-5
    case taskDeferCount           // number of deferrals
    case taskEstimatedMinutes     // duration
    case taskIsDoFirst            // boolean
    
    // Time context
    case dayOfWeek                // 1 (Mon) - 7 (Sun)
    case hourOfDay                // 0-23
    case isBeforeLunch            // boolean (before 12:00)
    case isAfterLunch             // boolean (after 13:00)
    case consecutiveMeetingHours  // float: hours of back-to-back meetings so far today
    case minutesSinceLastBreak    // int
    
    // User state
    case moodContext              // "easy_wins", "avoidance", "deep_focus"
    case totalTasksToday          // int
    case completedTasksToday      // int
    case currentEnergyEstimate    // "high", "medium", "low" (inferred from time + patterns)
    
    // Calendar context
    case hasUpcomingMeeting       // boolean: meeting within next 30 min
    case freeTimeRemainingMinutes // int: unscheduled minutes left in work day
}

enum TriggerOperator: String, Codable, Sendable {
    case equals          // ==
    case notEquals       // !=
    case greaterThan     // >
    case lessThan        // <
    case greaterOrEqual  // >=
    case lessOrEqual     // <=
    case contains        // string contains
    case inSet           // value is in a comma-separated set
    case notInSet        // value is not in a set
}
```

### 2.3 Rule Actions

```swift
enum RuleAction: Codable, Sendable {
    case schedulePreference(SchedulePreference)    // influence when a task is scheduled
    case scoringModifier(ScoringModifier)           // influence PlanningEngine scoring
    case alertTrigger(AlertConfig)                  // trigger a proactive alert
    case briefingInclusion(BriefingNote)            // include a note in the morning briefing
    case blockTimeSlot(TimeSlotBlock)               // suggest blocking a time range
}

struct SchedulePreference: Codable, Sendable {
    let preferredDayOfWeek: Int?       // nil = any day
    let preferredHourStart: Int?       // nil = any time
    let preferredHourEnd: Int?
    let avoidDayOfWeek: Int?           // day to avoid scheduling this type
    let avoidHourStart: Int?
    let avoidHourEnd: Int?
    let reason: String                 // "You complete HR tasks 40% faster on Tuesday mornings"
}

struct ScoringModifier: Codable, Sendable {
    let scoreAdjustment: Int           // added to PlanningEngine composite score
    let reason: String                 // human-readable explanation
}

struct AlertConfig: Codable, Sendable {
    let alertType: String              // "avoidance_warning", "energy_warning", "deadline_risk"
    let message: String                // "You've deferred this 4 times — what's blocking you?"
    let priority: AlertPriority        // how urgently to surface
    let cooldownMinutes: Int           // don't fire again for this many minutes
}

enum AlertPriority: String, Codable, Sendable {
    case low        // menu bar badge only
    case medium     // notification
    case high       // interrupt current flow
}

struct BriefingNote: Codable, Sendable {
    let noteType: String               // "pattern_observation", "avoidance_probe", "celebration"
    let content: String                // what to include in the morning briefing
}

struct TimeSlotBlock: Codable, Sendable {
    let dayOfWeek: Int                 // 1-7
    let startHour: Int                 // 0-23
    let endHour: Int
    let label: String                  // "Deep work", "Recovery", "Admin batch"
    let reason: String
}
```

```swift
struct RuleException: Codable, Sendable {
```

### 2.5 Rule Status Lifecycle

```swift
enum RuleStatus: String, Codable, Sendable {
    case proposed       // generated by reflection engine, not yet active
    case active         // confirmed (by time or user acceptance), actively applied
    case suspended      // temporarily disabled (user override or poor performance)
    case deprecated     // permanently retired
```

## 3. Rule Lifecycle

### 3.1 Generation (Proposed)

The reflection engine generates rules from semantic facts. A rule is always grounded in one or more semantic facts.

```
Semantic Fact: "Defers HR tasks on Monday mornings (confidence: 0.85)"
    ↓ Reflection Engine
Procedural Rule: "IF task.category == HR AND dayOfWeek == 1 THEN schedule for Tuesday 10am"
    status: proposed
    confidence: 0.60 (lower than source fact — rule is untested)
    sourceFactIds: [fact_uuid]
```

**Generation criteria:**
- The source semantic fact must have `confidence >= 0.50` and `status == .active`
- No existing active rule already covers the same trigger conditions
- The reflection engine must be able to formulate a specific, actionable rule (not a vague suggestion)

### 3.2 Activation

A proposed rule becomes active through one of two paths:

**Path A: Auto-activation (time-based)**
- Rule has been proposed for >= 3 days
- No user rejection during that period
- Source semantic facts still active with confidence >= 0.50

**Path B: User acceptance**
- Morning briefing presents the proposed rule: "I've noticed you defer HR tasks on Mondays. Should I start scheduling them for Tuesdays instead?"
- User confirms → immediate activation

**On activation:**
```swift
rule.status = .active
rule.activatedAt = Date()
```

### 3.3 Active Operation

While active, the rule is evaluated on every relevant trigger context. The evaluation engine checks:
1. Are all `triggerConditions` met?
2. Are any `exceptions` applicable?
3. Is the rule not in cooldown (for alert-type rules)?

If all conditions pass and no exceptions apply, the rule fires and its action is applied.

### 3.4 Suspension

A rule is suspended when:
- The user explicitly overrides it (e.g., manually schedules an HR task on Monday despite the rule suggesting Tuesday)
- The rule's `successRate` drops below 0.40 after >= 5 activations
- The source semantic fact transitions to `contradicted` or `deprecated`

```swift
rule.status = .suspended
// Suspended rules can be reactivated if conditions change
// They are not deleted — they retain their activation history
```

### 3.5 Deprecation

A rule is deprecated (permanently retired) when:
- Suspended for >= 30 days without reactivation
- Source semantic facts are all deprecated
- User explicitly says "stop doing this"
- `failureCount / activationCount > 0.7` after >= 10 activations (70%+ failure rate)

```swift
rule.status = .deprecated
rule.deprecatedAt = Date()
rule.deprecationReason = "Source fact deprecated: executive no longer defers HR tasks on Mondays"
```

## 4. How Rules Constrain Scheduling

### 4.1 Integration with PlanningEngine

The existing `PlanningEngine` already accepts `[BehaviourRule]` in its `PlanRequest`. Procedural rules map to this interface:

```swift
func toEngineBehaviourRule() -> BehaviourRule? {
    guard status == .active else { return nil }
    
    switch action {
    case .scoringModifier(let modifier):
        return BehaviourRule(
            ruleKey: id.uuidString,
            ruleType: "category_pref",
            ruleValueJson: try! JSONEncoder().encode(modifier),
            confidence: confidence
        )
    case .schedulePreference(let pref):
        return BehaviourRule(
            ruleKey: id.uuidString,
            ruleType: "timing",
            ruleValueJson: try! JSONEncoder().encode(pref),
            confidence: confidence
        )
    default:
        return nil  // alert and briefing rules don't affect scoring
    }
}
```

### 4.2 Score Impact Calculation

The PlanningEngine applies rule-based score adjustments in its existing `behaviourBump` calculation. The procedural rule's `confidence` acts as a weight:

```
effectiveScoreAdjustment = rule.scoreAdjustment * rule.confidence
```

A rule with `scoreAdjustment = +50` and `confidence = 0.85` contributes `+42` to the composite score.

### 4.3 Time Slot Influence

For `SchedulePreference` rules, the `TimeSlotAllocator` (existing) is extended to:
1. Check all active rules with `schedulePreference` actions
2. For each task being placed, check if any rules apply to its category/bucket
3. Prefer slots within the rule's `preferredHourStart..preferredHourEnd`
4. Avoid slots within the rule's `avoidHourStart..avoidHourEnd`
5. If no preferred slot is available, fall back to the default allocation

## 5. How Rules Shape Morning Briefings

### 5.1 Briefing Note Rules

Rules with `action == .briefingInclusion` inject notes into the morning session. The morning director (Opus) receives these as structured context alongside the semantic profile:

```json
{
  "activeRules": [
    {
      "ruleText": "Probe HR avoidance on Mondays",
      "action": {
        "briefingInclusion": {
          "noteType": "avoidance_probe",
          "content": "The executive has 2 HR tasks from last week. Historical pattern: defers HR on Mondays 80% of the time. Consider asking about these specifically."
        }
      },
      "confidence": 0.82,
      "activationCount": 12,
      "successRate": 0.75
    }
  ]
}
```

The morning director uses these as inputs to its intelligence briefing — not as scripts to read verbatim.

### 5.2 Proposed Rule Presentation

Proposed rules are surfaced in the morning briefing for user validation:

"I've noticed a pattern: you tend to push financial review tasks past their due date, then complete them in a rush on Fridays. Would it help if I blocked Thursday 2-4pm for financial review each week?"

The user's response (accept/reject/modify) is recorded as an episode and feeds back into the rule lifecycle.

## 6. How Rules Trigger Alerts

### 6.1 Alert Evaluation Loop

A lightweight evaluation loop runs every 5 minutes while the app is active:

```swift
actor AlertEvaluator {
    func evaluate(
        activeRules: [ProceduralRule],
        currentContext: TriggerContext
    ) -> [Alert] {
        activeRules
            .filter { $0.action.isAlertTrigger }
            .filter { meetsAllConditions($0.triggerConditions, context: currentContext) }
            .filter { !hasException($0.exceptions, context: currentContext) }
            .filter { !isInCooldown($0) }
            .map { buildAlert(from: $0) }
    }
}
```

### 6.2 TriggerContext

```swift
struct TriggerContext: Sendable {
    let currentTime: Date
    let dayOfWeek: Int
    let hourOfDay: Int
    let consecutiveMeetingHours: Float
    let minutesSinceLastBreak: Int
    let totalTasksToday: Int
    let completedTasksToday: Int
    let currentTask: TimedTask?
    let upcomingMeetingInMinutes: Int?
    let freeTimeRemainingMinutes: Int
    let moodContext: MoodContext?
}
```

The context is assembled from the current app state — DataStore for tasks, CalendarSyncService for meetings, FocusTimer for break tracking.

### 6.3 Cooldown Enforcement

After an alert fires, it enters cooldown:
```swift
private var cooldownExpiry: [UUID: Date] = [:]

func isInCooldown(_ rule: ProceduralRule) -> Bool {
    guard let expiry = cooldownExpiry[rule.id] else { return false }
    return Date() < expiry
}

func recordActivation(_ rule: ProceduralRule, cooldownMinutes: Int) {
    cooldownExpiry[rule.id] = Date().addingTimeInterval(TimeInterval(cooldownMinutes * 60))
}
```

## 7. Rule Conflict Resolution

### 7.1 Conflict Types

| Conflict | Example | Resolution |
|----------|---------|------------|
| **Contradictory scheduling** | Rule A: "Schedule HR before 11am." Rule B: "No tasks before 10am on Mondays." | Priority: rule with higher `confidence * successRate`. If tie: newer rule wins. |
| **Contradictory scoring** | Rule A: "+30 for client tasks." Rule B: "-20 for client tasks before lunch." | Additive: both apply. Net effect: +10 for client tasks before lunch. Rules are not exclusive. |
| **Contradictory alerts** | Rule A: "Alert if meetings > 3hrs." Rule B: "Don't alert during focus sessions." | Exception-based: Rule B is an exception on Rule A. The reflection engine should learn to add it as a `RuleException` on Rule A. |
| **Redundant rules** | Rule A and Rule B have identical trigger conditions and similar actions. | Merge: keep the one with higher activation count. Deprecate the other. |

### 7.2 Resolution Algorithm

```swift
func resolveConflicts(applicableRules: [ProceduralRule]) -> [ProceduralRule] {
    // 1. Group by action type
    let grouped = Dictionary(grouping: applicableRules, by: { $0.action.actionType })
    
    var resolved: [ProceduralRule] = []
    
    for (actionType, rules) in grouped {
        switch actionType {
        case .schedulePreference:
            // For scheduling conflicts: highest confidence * successRate wins
            let ranked = rules.sorted { effectiveWeight($0) > effectiveWeight($1) }
            resolved.append(ranked[0])  // only the strongest scheduling rule applies
            
        case .scoringModifier:
            // All scoring modifiers are additive — no conflict, all apply
            resolved.append(contentsOf: rules)
            
        case .alertTrigger:
            // All alerts fire independently unless one is an exception to another
            resolved.append(contentsOf: rules)
            
        case .briefingInclusion:
            // All briefing notes are included (Opus decides how to use them)
            resolved.append(contentsOf: rules)
            
        case .blockTimeSlot:
            // For time slot conflicts: merge overlapping blocks, highest confidence wins on overlap
            resolved.append(contentsOf: mergeTimeSlots(rules))
        }
    }
    
    return resolved
}

func effectiveWeight(_ rule: ProceduralRule) -> Float {
    let successRate = rule.activationCount > 0 
        ? Float(rule.successCount) / Float(rule.activationCount) 
        : 0.5  // neutral if untested
    return rule.confidence * successRate
}
```

## 8. Rule Effectiveness Tracking

### 8.1 What Counts as Success or Failure

| Rule action type | Success signal | Failure signal |
|-----------------|----------------|----------------|
| `schedulePreference` | Task completed in the suggested time slot without deferral | Task deferred from suggested slot, or completed in a different slot |
| `scoringModifier` | Task completed without user overriding the priority | User overrides the priority in the opposite direction of the modifier |
| `alertTrigger` | User acknowledges alert and takes action within 30 min | User dismisses alert without action, or explicitly marks as unhelpful |
| `briefingInclusion` | Morning interview references the briefing topic, user engages | Morning interview skips the topic entirely, or user says "that's not relevant" |
| `blockTimeSlot` | User uses the blocked time for the intended purpose | User schedules something else in the blocked time, or has no activity |

### 8.2 Outcome Recording

When a rule activates, the system records the activation and monitors for the outcome signal:

```swift
struct RuleActivation: Codable, Sendable {
    let id: UUID
    let ruleId: UUID
    let activatedAt: Date
    let triggerContext: TriggerContext
    let outcome: RuleOutcome?
    let outcomeRecordedAt: Date?
}

enum RuleOutcome: String, Codable, Sendable {
    case success
    case failure
    case neutral    // couldn't determine outcome
    case userOverride  // user explicitly went against the rule
}
```

Outcomes are determined by the nightly reflection engine, which looks at what happened after each activation.

### 8.3 Performance Metrics

```swift
extension ProceduralRule {
    var successRate: Float {
        guard activationCount > 0 else { return 0.5 }
        return Float(successCount) / Float(activationCount)
    }
    
    var isEffective: Bool {
        activationCount >= 5 && successRate >= 0.50
    }
    
    var isUnderperforming: Bool {
        activationCount >= 5 && successRate < 0.40
    }
    
    var needsMoreData: Bool {
        activationCount < 5
    }
}
```

## 9. Deprecation Criteria

A rule is deprecated when ANY of the following are true:

| Criterion | Threshold | Rationale |
|-----------|-----------|-----------|
| Failure rate | > 70% after 10+ activations | Rule is wrong more than it's right |
| Inactivity | Not activated for 60 days | Trigger conditions never occur — rule is irrelevant |
| Source fact deprecated | All `sourceFactIds` point to deprecated facts | The understanding that generated this rule is no longer valid |
| Sustained suspension | Suspended for 30+ days | If it was suspended and nobody reactivated it, it's not needed |
| User explicit rejection | User says "stop this rule" | Hard override, immediate deprecation |
| Confidence floor | Confidence decays below 0.15 | Rule has been contradicted and unreinforced into irrelevance |

### 9.1 Confidence Decay for Rules

Rules decay faster than semantic facts because they are actionable — a wrong rule causes harm, a slightly outdated fact is merely imprecise.

```swift
func applyRuleDecay(rule: inout ProceduralRule, currentDate: Date) {
    guard let lastActivated = rule.lastActivated else {
        // Never activated — decay from proposed date
        let daysSinceProposed = currentDate.timeIntervalSince(rule.proposedAt) / 86400
        if daysSinceProposed > 30 {
            rule.confidence *= 0.95  // 5% per day after 30 days without activation
        }
        return
    }
    
    let daysSinceActivation = currentDate.timeIntervalSince(lastActivated) / 86400
    guard daysSinceActivation > 14 else { return }  // 14-day grace period
    
    let dailyDecay: Float = 0.005  // ~50% after 140 days without activation
    let decayDays = Float(daysSinceActivation - 14)
    rule.confidence = max(rule.confidence - (dailyDecay * decayDays), 0.05)
}
```

## 10. Bridging to Existing BehaviourRule

The existing `BehaviourRule` struct in `PlanningEngine.swift` is the current consumer of procedural knowledge:

```swift
struct BehaviourRule: Sendable {
    let ruleKey: String
    let ruleType: String  // "ordering" | "threshold" | "timing" | "category_pref"
    let ruleValueJson: Data
    let confidence: Float
}
```

**Bridge strategy:** `ProceduralRule` is the authoritative store. At plan-time, active procedural rules with scheduling or scoring actions are mapped to `BehaviourRule` structs via `toEngineBehaviourRule()` (Section 4.1). The PlanningEngine does not need to change — it already consumes `BehaviourRule` by design.

Over time, all hardcoded or manually-configured `BehaviourRule` entries are replaced by learned `ProceduralRule` entries. The PlanningEngine remains a pure function — it takes rules as input and produces a scored plan. It does not care where the rules came from.

## 11. Acceptance Criteria

1. **Rule generation from semantic fact** — Create a semantic fact with confidence 0.70. Run the reflection engine. A procedural rule is generated in `proposed` status with correct `sourceFactIds`, structured `triggerConditions`, and a specific `action`.

2. **Auto-activation works** — Propose a rule. Advance time 3 days without user rejection. Rule transitions to `active` automatically.

3. **User acceptance activates immediately** — Propose a rule. User confirms in morning interview. Rule transitions to `active` with `activatedAt == now`.

4. **Rule fires correctly** — Create an active rule with `triggerConditions`: `taskCategory == HR AND dayOfWeek == 2`. Build a `TriggerContext` matching those conditions. Rule fires. Build a context where `dayOfWeek == 3`. Rule does not fire.

5. **Exception handling works** — Add an exception to the rule: `taskSender contains "CEO"`. Build a context matching the main conditions AND the exception. Rule does not fire.

6. **Scoring modifier integrates with PlanningEngine** — Active rule with `scoringModifier(+30)` and `confidence 0.8`. PlanTask matching the rule's conditions receives +24 (`30 * 0.8`) in `behaviourBump`.

7. **Alert cooldown works** — Rule fires an alert with `cooldownMinutes = 60`. Immediately evaluate again. Rule does not fire. Advance time 61 minutes. Rule fires again.

8. **Conflict resolution works** — Two active scheduling preference rules conflict on the same task type. The one with higher `confidence * successRate` wins. Both scoring modifier rules on the same task apply additively.

9. **Effectiveness tracking works** — Rule activates 10 times. 7 successes, 3 failures. `successRate == 0.7`. Rule remains active. Change to 3 successes, 7 failures. `successRate == 0.3`. Rule transitions to `suspended`.

10. **Deprecation works** — Suspend a rule. Advance 30 days. Rule transitions to `deprecated` with `deprecationReason` populated. Deprecated rules are never evaluated by the alert loop or returned by the PlanningEngine bridge.

11. **Source fact linkage** — Deprecate a semantic fact. All procedural rules with that fact in `sourceFactIds` are flagged for review. If the fact was the sole source, the rule transitions to `suspended`.

12. **Rule history is auditable** — For any active rule, query its full history: when proposed, when activated, every activation with outcome, every exception added, confidence trajectory. All data is present and consistent.
