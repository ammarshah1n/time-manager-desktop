# Month-Over-Month Intelligence Report Spec

**Status:** Implementation-ready
**Layer:** 4 (Intelligence Delivery) — consumes Layer 2 (Memory Store) + ModelVersioningService
**Owner:** IntelligenceReportGenerator
**Depends on:** ModelVersioningService, UserIntelligenceStore, PlanStore, QueryEngine, ReflectionEngine

---

## 1. Purpose

Prove that the system is getting smarter. Every 30 days, Timed generates a comprehensive intelligence report that quantifies the depth, accuracy, and value of its cognitive model. This is the document that justifies the product's existence — it shows the executive (and themselves) that month 6 is incomparably more intelligent than month 1.

The report is also the system's self-assessment mechanism. If the metrics are flat or declining, something is wrong — either the system isn't learning, or the executive isn't using it enough to feed the loop.

---

## 2. Report Cadence

| Report | Generated At | Covers |
|---|---|---|
| 30-Day Report | Day 30, 03:00 | Day 1–30 |
| 60-Day Report | Day 60, 03:00 | Day 31–60 (with cumulative context) |
| 90-Day Report | Day 90, 03:00 | Day 61–90 (with cumulative context) |
| 180-Day Report | Day 180, 03:00 | Day 91–180 (with cumulative context) |
| Monthly (ongoing) | 1st of each month, 03:00 | Previous calendar month |

Each report is generated by the IntelligenceReportGenerator using Opus (this is a high-value synthesis task — no model compromise). The report is stored in CoreData and presented in the app's Intelligence tab.

## 3. Report Sections

### 3.1 Executive Summary

A 3–5 sentence overview written by Opus that captures the state of the model in plain language. This is the only section some executives will read.

**Examples by maturity:**

**Day 30:**
"After 30 days, I know 67 facts about how you operate with an average confidence of 0.52. I've identified 12 behavioural patterns, 3 of which are confirmed. My time estimates are 34% more accurate than your self-estimates. The biggest insight so far: you consistently underallocate time to strategic work on Mondays, spending 78% of the day in reactive mode. I'm still learning — ask me again at day 60."

**Day 90:**
"At 90 days, my model contains 156 facts (avg confidence 0.71) and 28 confirmed patterns including 4 second-order patterns. My day plans now match your actual behaviour 68% of the time, up from 41% at day 30. I've detected a recurring cycle: you defer difficult conversations for an average of 6.2 days, then handle them in a burst on Fridays. This pattern has cost you an estimated 12 hours of accumulated anxiety and context-switching over the last month."

**Day 180:**
"Six months in. 234 facts, 0.79 average confidence, 41 confirmed patterns (9 second-order). My predictions about your energy levels are accurate to within 15 minutes of your actual performance peaks. The model has evolved most dramatically in understanding your relationship dynamics — I can now predict which stakeholder relationships need attention before you notice the drift yourself. Compared to month 1, this is a qualitatively different system."

### 3.2 Model Specificity Metrics

These metrics quantify how much the system knows and how confident it is.

```swift
struct ModelSpecificityMetrics {
    // Knowledge breadth
    let semanticFactCount: Int
    let semanticFactCountDelta: Int          // change from last report
    let factsByCategory: [String: Int]       // work_style: 34, preferences: 28, relationships: 41...
    
    // Knowledge confidence
    let averageConfidence: Float
    let averageConfidenceDelta: Float
    let confidenceDistribution: ConfidenceDistribution
    let highConfidenceFacts: Int             // confidence > 0.8
    let highConfidenceFactsDelta: Int
    
    // Knowledge depth
    let factsWithMultipleSources: Int        // reinforced by 3+ episodic memories
    let averageReinforcementCount: Float
    
    // Composite specificity score
    let specificityScore: Float              // factCount * avgConfidence / 100 (normalised)
    let specificityScoreDelta: Float
}
```

**Trend chart: Model Specificity Over Time**
- X-axis: weeks since install
- Y-axis: specificity score
- Expected curve: logarithmic growth — fast initial gain, slower as the model fills in

### 3.3 Rule Effectiveness Metrics

Measures whether the system's learned rules actually improve outcomes when applied.

```swift
struct RuleEffectivenessMetrics {
    let totalActiveRules: Int
    let totalActiveRulesDelta: Int
    
    // Effectiveness tracking
    let rulesWithOutcomeData: Int            // rules that have been applied enough to measure
    let effectiveRules: Int                  // rules where application improved outcomes
    let ineffectiveRules: Int                // rules where application had no measurable effect
    let counterproductiveRules: Int          // rules where application worsened outcomes
    let effectivenessRate: Float             // effective / rulesWithOutcomeData
    let effectivenessRateDelta: Float
    
    // Rule lifecycle
    let rulesCreatedThisPeriod: Int
    let rulesRetiredThisPeriod: Int
    let rulesSuspendedThisPeriod: Int
    let averageRuleLifespan: Int             // days from creation to retirement (for retired rules)
    
    // Top performing rules
    let topRules: [RulePerformanceSummary]   // top 5 by effectiveness
    let worstRules: [RulePerformanceSummary] // bottom 3 — candidates for retirement
}

struct RulePerformanceSummary {
    let ruleID: UUID
    let ruleText: String
    let timesApplied: Int
    let successRate: Float
    let estimatedTimeSaved: Int              // minutes
}
```

**How rule effectiveness is measured:**

A rule is "effective" if the outcome when the rule was followed is measurably better than the baseline outcome when it was not followed. Measurement:
- Scheduling rules: Was the task completed in the window the rule specified? Was the quality (completion without revision) higher?
- Avoidance rules: When the system flagged an avoidance pattern, did the executive act on the flagged item sooner than their historical average?
- Priority rules: When a rule promoted a task, was the task completed (not deferred)?

### 3.4 Prediction Accuracy Metrics

How accurate are the system's predictions about the executive?

```swift
}
```

**Trend chart: Prediction Accuracy Over Time**
- X-axis: weeks since install
- Y-axis: composite prediction score (weighted average of all prediction metrics)
- Expected curve: steep improvement weeks 2–8 as EMA and Thompson sampling calibrate, then gradual improvement as the model deepens

## 4. Report Generation Process

### 4.1 Data Collection

At report generation time (03:00 on report day), the IntelligenceReportGenerator:

1. Loads the current model snapshot and the snapshot from the start of the reporting period
2. Computes the ModelDiff between them
3. Queries the PlanStore for all day plans in the period
4. Queries the QueryEngine logs for all queries in the period
5. Queries the ReflectionEngine logs for all reflection cycles in the period
6. Aggregates all metrics defined in Section 3

### 4.2 Narrative Generation (Opus)

The raw metrics are passed to Opus with the following prompt:

```
You are the intelligence report writer for Timed. Write a comprehensive but concise report for the executive.

## Report Period
{period_start} to {period_end} (Day {day_number} since install)

## Raw Metrics
{all_metrics_json}

## Previous Report Summary (for context)
{previous_report_executive_summary}

## Model Diff Summary
{diff_summary}

## Instructions
1. Write an executive summary (3-5 sentences). Lead with the single most impressive metric. Be specific — use numbers.
2. For each metric section, write 2-3 sentences of analysis. Not just "X increased" — explain WHY and what it means.
3. Identify the single most significant model evolution this period. What did the system learn that it couldn't have known before?
4. Identify the biggest gap or weakness in the current model. What is the system still bad at?
5. If usage is low, say so directly: "I can't get smarter if you don't use me. Morning sessions this month: {count}/{workdays}."
6. End with a forward-looking statement: what the system expects to learn next month.

Tone: Direct, data-driven, no fluff. This executive has zero patience for padding. Every sentence must carry information.
```

### 4.3 Trend Chart Data

The generator produces time-series arrays for each trend metric (see Section 3). The UI layer renders these as line charts. Chart descriptions are included in the report text for accessibility:

- "Model Specificity: Started at 12.3 on day 1, currently at 67.8. Growth was steepest between days 14–45, then tapered to a steady climb."
- "Prediction Accuracy: Time estimation MAE was 42 minutes at day 7. Now 18 minutes. The biggest improvement came after week 3 when EMA coefficients stabilised."

## 5. Report Output Format

```swift
struct IntelligenceReport: Codable, Identifiable {
    let id: UUID
    let reportType: ReportType          // .day30, .day60, .day90, .day180, .monthly
    let periodStart: Date
    let periodEnd: Date
    let daysSinceInstall: Int
    let generatedAt: Date
    
    // Sections
    let executiveSummary: String
    let modelSpecificity: ModelSpecificityMetrics
    let modelSpecificityNarrative: String
    let ruleEffectiveness: RuleEffectivenessMetrics
    let ruleEffectivenessNarrative: String
    let predictionAccuracy: PredictionAccuracyMetrics
    let predictionAccuracyNarrative: String
    let patternDetection: PatternDetectionMetrics
    let patternDetectionNarrative: String
    let retrievalPrecision: RetrievalMetrics
    let retrievalPrecisionNarrative: String
    let usage: UsageMetrics
    let usageNarrative: String
    
    // Trends (for chart rendering)
    let trends: [ModelTrend]
    
    // Highlights
    let biggestEvolution: String         // single most significant model change
    let biggestGap: String               // single biggest weakness
    let forwardLook: String              // what the system expects to learn next
    
    // Comparison to previous report
    let previousReportID: UUID?
    let improvementScore: Float          // composite improvement vs previous report (0.0–1.0)
}

enum ReportType: String, Codable {
    case day30
    case day60
    case day90
    case day180
    case monthly
}
```

## 6. Proving the System Gets Smarter

This is the report's fundamental purpose. The following metrics, tracked over time, constitute proof of compounding intelligence:

### 6.1 Compounding Evidence Chain

| Metric | Month 1 (Expected) | Month 3 (Expected) | Month 6 (Expected) | What It Proves |
|---|---|---|---|---|
| Semantic fact count | 50–80 | 140–180 | 200–250 | Knowledge breadth grows |
| Average confidence | 0.40–0.50 | 0.60–0.70 | 0.75–0.85 | Knowledge depth grows |
| Time estimation MAE | 35–45 min | 15–25 min | 10–15 min | Predictions improve |
| Plan adherence rate | 35–45% | 55–65% | 65–75% | Plans match reality |
| First-order patterns | 5–10 | 25–35 | 35–50 | Behaviour understanding grows |
| Second-order patterns | 0 | 3–6 | 6–12 | Meta-understanding emerges |
| Rule effectiveness rate | N/A (insufficient data) | 55–65% | 70–80% | Rules work |
| Query confidence avg | 0.35–0.45 | 0.55–0.65 | 0.70–0.80 | Answers improve |

### 6.2 The "Aha" Moments

Beyond metrics, the report should capture qualitative "aha" moments — the first time the system produces an insight that surprises the executive. These are logged when:
- The executive gives a thumbs-up on a query response (rare event — executives don't click things unless genuinely impressed)
- The morning session produces a pattern callout that the executive verbally confirms ("Yeah, that's exactly right")
- A procedural rule prevents a mistake (e.g., the system flags an avoidance pattern and the executive acts on it)

Each "aha" moment is stored as a high-importance episodic memory and referenced in the report.

## 9. Edge Cases

| Scenario | Behaviour |
|---|---|
| Day 30 with < 10 days of data (low usage) | Generate the report but lead with: "This report covers 30 days, but I only have meaningful data from [X] days. The metrics below are preliminary. Use me more and the next report will be dramatically better." |
| All metrics declining | Lead with honesty: "My model quality declined this month. [Reason if detectable: e.g., low usage, conflicting signals, role change]. Here's what I need to get back on track: [specific actions]." |
| Executive never reads reports | Track report-open events. If 3 consecutive reports go unopened, reduce report generation to quarterly and mention in morning session: "I've been generating monthly intelligence reports that you haven't opened. I'm switching to quarterly. Let me know if you want monthly back." |
| Report generation fails (Opus timeout) | Retry once after 30 minutes. If second attempt fails, generate a metrics-only report without Opus narrative. Flag: "The narrative section of this report couldn't be generated. Raw metrics are available." |
| First report (Day 30) | Include a "baseline comparison" section that compares current state to the cold-start state (day 1 snapshot). This shows the executive how much the system has already learned, even if absolute metrics are modest. |

## 10. Anti-Gaming

The report must not be gameable — neither by the system inflating its own metrics nor by the executive artificially creating data.

- Prediction accuracy is measured against ACTUAL outcomes, not self-reported outcomes
- Plan adherence is measured against observed behaviour (app focus, task completion signals), not plan checkboxes
- Query confidence is derived from retrieval scores, not from the synthesis model's self-assessment
- If the system detects that metrics are being inflated by trivial data (e.g., dozens of identical short focus sessions padding completion counts), it flags this in the report and excludes the trivial data from metric computation
