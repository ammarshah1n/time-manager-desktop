# 02 — Memory System Specification

## Overview

Three-tier memory architecture inspired by MemGPT (UC Berkeley, 2023) and
Stanford Generative Agents (Park et al., 2023).

The memory system is the foundation of compounding intelligence. Without it,
Timed is just another planning app. With it, month 6 is incomparably more
intelligent than month 1.

## Memory Tiers

### Tier 1: Episodic Memory
Raw, timestamped records of what happened. Written by Signal Ingestion.
Never modified after creation — only read and eventually consolidated.

**Schema:**
```swift
struct EpisodicMemory: Identifiable, Codable {
    let id: UUID
    let timestamp: Date
    let source: SignalSource          // .email, .calendar, .voice, .behaviour, .task
    let category: MemoryCategory      // .communication, .work, .meeting, .avoidance, .decision
    let content: String               // natural language description
    let rawData: [String: AnyCodable] // structured signal data
    let importanceScore: Double       // 0.0-1.0, assigned at write time by Haiku
    let embedding: [Float]?           // Jina AI 1024-dim vector, computed async
}
```

**Examples:**
- "Responded to CFO email in 45 seconds (typical: 8 minutes)"
- "Opened board deck at 10:23, closed at 10:24 without editing (3rd time this week)"
- "Morning interview: mentioned 'Meridian deal' unprompted with hesitant tone"
- "Deferred 'Head of Sales 1:1' for 4th consecutive day"

### Tier 2: Semantic Memory
Distilled facts about the person. Written by the Reflection Engine during
synthesis cycles. Can be updated/versioned as understanding deepens.

**Schema:**
```swift
struct SemanticFact: Identifiable, Codable {
    let id: UUID
    let createdAt: Date
    var updatedAt: Date
    let category: SemanticCategory    // .chronotype, .relationship, .preference, .capacity, .pattern
    let fact: String                  // natural language
    let confidence: Double            // 0.0-1.0, increases with confirming evidence
    let evidenceCount: Int            // number of episodic memories supporting this
    let sourceMemoryIDs: [UUID]       // traceability to episodic evidence
    let embedding: [Float]?
}
```

**Examples:**
- "Peak analytical performance window: 9:30-11:30 (confidence: 0.87, n=34)"
- "Relationship with CTO: response latency trending up 3x over 30 days"
- "Post-board meeting recovery: 72 hours of reduced strategic output"
- "Estimation accuracy for 'reply' tasks: consistently 40% under-estimated"

### Tier 3: Procedural Memory
Operating rules extracted from patterns. Generated by the Reflection Engine.
These directly constrain scheduling, scoring, and morning session content.

**Schema:**
```swift
struct ProceduralRule: Identifiable, Codable {
    let id: UUID
    let createdAt: Date
    var updatedAt: Date
    let trigger: RuleTrigger          // when this rule activates
    let action: RuleAction            // what the system should do
    let reasoning: String             // why this rule exists (traceable to insights)
    let confidence: Double            // 0.0-1.0
    let activationCount: Int          // how many times this rule has fired
    let sourceInsightIDs: [UUID]      // traceability to synthesis
}
```

**Examples:**
- TRIGGER: task involves confrontational conversation
  ACTION: add 6.2 day average deferral warning, surface in morning session
- TRIGGER: Thursday afternoon, back-to-back meetings > 2.5 hours
  ACTION: flag any people decisions as "high revision risk"
- TRIGGER: 72 hours post-board meeting
  ACTION: do not schedule strategic document work, suggest recovery tasks

## Core Memory (Always-in-Context)

A small, curated subset that is always loaded into any AI interaction.
Analogous to MemGPT's "core memory" that fits in the context window.

```swift
struct CoreMemorySnapshot: Codable {
    let executiveName: String
    let role: String
    let currentPriorities: [String]          // max 5
    let activeProjects: [String]             // max 10
    let keyRelationships: [RelationshipSummary]  // max 15
    let chronotypeProfile: ChronotypeProfile
    let currentStressIndicators: [String]
    let activeProceduralRules: [ProceduralRule]  // max 20 highest-confidence
    let lastReflectionDate: Date
}
```

## Retrieval Architecture

### Three-Axis Scoring (Park et al., 2023)
```
score = α × recency + β × importance + γ × relevance
```

- **Recency:** Exponential decay (0.995 hourly decay rate)
- **Importance:** LLM-assigned at memory write time (0.0-1.0)
- **Relevance:** Cosine similarity between memory embedding and query embedding

### Dynamic Weight Adjustment (ACAN, 2025)
Retrieval weights (α, β, γ) are NOT static. They adapt based on query context:
- Morning briefing → high recency weight (what happened recently?)
- Monthly pattern report → high importance weight (what matters most?)
- Specific query → high relevance weight (what's most similar?)
- Avoidance detection → high recency + importance (recent significant deferrals)

### Retrieval Precision > Volume
The morning session should surface 3-5 memories/patterns with high signal,
not 50 facts. Context pollution (too many memories loaded) degrades intelligence
quality. Maximum precision, not maximum recall.

## Memory Lifecycle

1. **Write:** Signal Ingestion creates episodic memory with importance score
2. **Embed:** Async job computes Jina AI embedding for semantic search
3. **Retrieve:** Reflection Engine or Delivery layer queries with three-axis scoring
4. **Consolidate:** Reflection Engine promotes episodic clusters → semantic facts
5. **Decay:** Old episodic memories decay in recency score (never deleted, just deprioritised)
6. **Update:** Semantic facts get confidence updates as new evidence arrives
7. **Generate:** Reflection Engine creates procedural rules from semantic patterns

## Memory Behaviour Tracking (OpenAI Context Engineering, 2026)

Three categories of personal data behaviour:
- **Stability:** Preferences that rarely change (chronotype, communication style)
- **Drift:** Gradual changes over time (shifting peak hours, evolving relationships)
- **Contextual variance:** Preferences that differ by context (Monday energy vs Friday)

The Reflection Engine must detect and track all three categories.

## Persistence Strategy

- **Local:** Episodic memories cached locally for offline access + fast retrieval
- **Remote:** All tiers synced to Supabase for durability and reflection engine access
- **Embeddings:** Stored alongside memories, indexed for cosine similarity search
- **No CoreData:** Continuing with DataStore actor (JSON local) + Supabase pattern
